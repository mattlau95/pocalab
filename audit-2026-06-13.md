# UX Audit Report — pocalab

> **Date:** 2026-06-13 · **Reviewed:** http://localhost:5176 (Vite dev), full source · **Tools run:** axe-core 4.11.4, pa11y, Lighthouse (performance/accessibility/best-practices), manual code review

---

## Executive summary

pocalab is well-structured for a single-page tool: data safety is solid, loading states are present on every async action, destructive confirmations exist, and semantic HTML is used throughout. The main issues are a silent export failure on mobile, two WCAG-AA color contrast violations caught by axe, and two undersized touch targets in the DeckCard component. LCP is 9.0s in dev mode — largely a Vite HMR artifact — but there is a real production concern: the Google Fonts stylesheet is synchronously render-blocking.

**4 blockers, 4 important, 4 polish.**

**Snapshot**

- Checklist items passing: **28 / 40**
- Core Web Vitals (dev server — inflate LCP, not representative of production): LCP **9.0s** · INP/TBT proxy **0ms** · CLS **0**
- Lighthouse scores: Performance **61** · Accessibility **92** · Best Practices **100**
- axe-core: **2** violations (color-contrast) · pa11y: **0** violations

> ⚠️ The LCP of 9.0s is measured against the un-bundled Vite dev server and is not representative of production. Run `vite build && vite preview` and re-measure before treating it as a real regression. The Google Fonts issue below *is* a real production concern.

## 🔴 P0 — Blockers

| # | Finding | Checklist ref | Evidence | Fix | Effort |
|---|---------|---------------|----------|-----|--------|
| 1 | Export error is never shown on mobile | §1 — system status (silent failure) | `App.tsx:347` — `exportError` renders only inside `.deck-actions--desktop`, which is `display:none` on `pointer:coarse`. Mobile deck-bar has no error path. | Add `{exportError && <p className="deck-actions__error">{exportError}</p>}` inside `.deck-bar__right` after the Download button in the deck-bar JSX | ⚡ |
| 2 | `.app-header__tagline` fails WCAG AA contrast | §7 — accessibility (color contrast) | axe-core: color-contrast violation. `App.css:47-51` — `opacity: 0.65` on `var(--foreground-muted)` reduces effective contrast below 4.5:1 on both light and dark palettes. | Remove `opacity: 0.65`; instead use a lighter token or a pre-blended hex that passes 4.5:1. E.g. in light mode use `#7A6558` directly (passes ~4.6:1 against `#FFFBF7`). | ⚡ |
| 3 | `.upload-zone__hint` fails WCAG AA contrast | §7 — accessibility (color contrast) | axe-core: color-contrast violation. `ImageUpload.css:38-42` — `opacity: 0.6` reduces effective contrast. Same root cause as #2. | Remove `opacity: 0.6`; use a pre-computed color that passes 4.5:1. | ⚡ |
| 4 | DeckCard action buttons below WCAG 2.2 Target Size minimum (24×24 px) | §8 — responsive & touch (target size) | `DeckCard.css:79` — `.deck-card__dl` is `20×20px`. `DeckCard.css:98-103` — `.deck-card__edit` height is ~20px (`font-size:11px; padding:2px 8px`). WCAG 2.2 SC 2.5.8 requires 24×24px minimum for all interactive targets. | Increase `.deck-card__dl` to `24×24px` min. Increase `.deck-card__edit` padding to at least `4px 8px` to reach ~24px height. | ⚡ |

## 🟡 P1 — Important

| # | Finding | Checklist ref | Evidence | Fix | Effort |
|---|---------|---------------|----------|-----|--------|
| 1 | Sticky header can obscure focused elements (Focus Not Obscured WCAG 2.2) | §7 — accessibility | `App.css:7-14` — `.app-header` is `position:sticky; top:0` (~64px tall). No `scroll-padding-top` is set, so keyboard Tab can bring an element into focus just below the header, where it's visually hidden. | Add `html { scroll-padding-top: 80px; }` to `index.css`. This pushes scroll-into-view targets below the sticky header. | ⚡ |
| 2 | Google Fonts CSS is synchronously render-blocking | §9 — performance | `index.html:8` — `<link rel="stylesheet" href="https://fonts.googleapis.com/...">` blocks first paint. Even with `display=swap` on font-face, the CSS file fetch itself is synchronous. Impacts production LCP. | Use the async stylesheet pattern: `<link rel="preload" href="..." as="style" onload="this.onload=null;this.rel='stylesheet'"><noscript><link rel="stylesheet" href="..."></noscript>`. | ⚡ |
| 3 | Background color `<label>` not programmatically associated with its `<input>` | §4 — forms & input | `CropEditor.tsx:273` — `<label className="control-label">Background</label>` has no `htmlFor`. The `<input type="color">` inside `.ctrl-swatch__input` has no `id`. Screen readers can find the input but won't announce its label. | Add `id="crop-bgcolor"` to the color input and `htmlFor="crop-bgcolor"` to the label. | ⚡ |
| 4 | Crop viewport drag-to-pan has no keyboard position alternative | §7 — accessibility (dragging movements WCAG 2.2) | `CropEditor.tsx:189-227` — react-easy-crop only exposes pan via pointer drag. WCAG 2.2 SC 2.5.7 requires a single-pointer (non-drag) alternative for all drag interactions. The Center pill partially mitigates (centers to 0,0) but arbitrary positioning is not keyboard-reachable. | At minimum: document that Center covers the 90% case, and add arrow-key nudge: listen for keydown on the viewport div and increment `crop.x`/`crop.y` by a small step. | 🔨 |

## 🟢 P2 — Polish

| # | Finding | Checklist ref | Evidence | Fix | Effort |
|---|---------|---------------|----------|-----|--------|
| 1 | Copy stepper (−/+) and crop ctrl-pills are below the 44px comfortable touch target | §8 — responsive & touch | `DeckCard.css:121` — `.copies-btn` is `26×26px`. `CropEditor.css:212-222` — `.ctrl-pill` height ~26px. Both exceed the 24px hard minimum but fall short of the 44px recommended touch target. | Increase to at least `32×32px` on `pointer:coarse` via media query. Or live with it — these are secondary controls and the DeckCard thumbs area is the primary interaction. | ⚡ |
| 2 | Remove card confirmation text doesn't identify the specific card | §2 — destructive actions | `DeckCard.tsx:38` — `window.confirm('Remove this card from your deck?')`. With multiple cards in the deck, the user can't tell which is about to be removed (the confirm fires before focus is obvious). | Add a card index or thumbnail reference to the message, or replace `window.confirm` with a small inline confirmation toggle ("Really remove? Yes / No") that appears on the card itself. | 🔨 |
| 3 | `app-header__count` badge has no ARIA context | §7 — accessibility | `App.tsx:306` — `<span className="app-header__count">{total} / {DECK_MAX_CARDS} cards</span>`. Screen readers will read the text but it lacks `aria-live="polite"` so changes (card added/removed) aren't announced. | Add `aria-live="polite"` to the span. | ⚡ |
| 4 | No visual feedback when a card is removed from the deck | §1 — system status | `App.tsx` — `removeCard` immediately filters the card out; the grid snaps. A CSS exit animation on `.deck-card` (e.g. `opacity: 0; scale: 0.9`) would confirm the action and prevent the "wait, did it work?" moment. | Add a short exit animation on `.deck-card` before calling `removeCard`, or use a CSS `@starting-style` transition. | 🔨 |

## What passed

- **Data safety** — `useBeforeUnload` fires when deck has cards or a crop is in progress; localStorage persists across refresh with migration-safe hydration; every destructive cancel asks for confirmation.
- **Loading states** — both PDF export and crop confirm show spinner and disable the button while async work is in flight. No frozen UI anywhere.
- **Error states** — export errors, crop errors, and file-validation errors all surface with human-readable messages. Empty deck shows a clear first-run upload prompt.
- **Semantic HTML** — real `<button>`, `<header>`, `<main>`, `<label>` throughout. No div soup.
- **Focus styles** — `:focus-visible { outline: 2px solid var(--primary) }` globally defined. Focus rings are visible and on-brand.
- **Reduced motion** — `@media (prefers-reduced-motion: reduce)` cuts all transitions to 0.01ms.
- **Destructive confirmations** — remove card and cancel-with-loss both confirm with specific, actionable text.
- **Decorative images** — icon PNGs correctly have `alt=""`.
- **Best Practices** — Lighthouse 100/100; no mixed content, no deprecated APIs.
- **CLS** — 0 (no layout shift).
- **TBT / INP proxy** — 0ms (no long JavaScript tasks blocking the main thread).
